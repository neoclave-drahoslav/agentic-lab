# ADR 001: Agent Instruction Build System

## Status

Proposed

## Context

The monorepo is used by multiple AI coding tools. Each tool expects
instructions in a different location or format. Directly editing those files
creates behavioral drift between agents.

The monorepo also contains multiple domains owned by different teams. Each team
needs to maintain its own domain boundaries, testing expectations, review
triggers, and agent-specific constraints without forking the global policy.

The team also wants to use different tools to generate or modify functional
projects from the same brief. Those tools should share the same agent roles,
policies, domain boundaries, and verification expectations.

## Decision

Use `.agents/` as the canonical source of truth and generate tool-specific
adapters from it.

Global instruction infrastructure is platform-owned:

- `.agents/source/**`
- `.agents/roles/**`
- `.agents/templates/**`
- `scripts/agents.mjs`

Domain instructions are team-owned:

- `.agents/domains/billing/**` by Team Revenue.
- `.agents/domains/identity/**` by Team Platform.
- `.agents/domains/frontend/**` by Team Experience.

Generated files include:

- `CLAUDE.md`
- `AGENTS.md`
- `.cursor/rules/agentic-engineering.mdc`
- `.github/copilot-instructions.md`
- `.github/instructions/*.instructions.md`
- domain-local `AGENTS.md` files

The canonical role roster is:

- Project Architect Agent.
- Implementation Agent.
- Test Agent.
- Review Agent.
- Integration Agent.

CI should run:

```bash
npm run agents:check
```

## Consequences

- Agent behavior is easier to govern across tools.
- Every tool receives the same role definitions and policy set.
- Domain teams can own their own context without forking global policy.
- Generated files must not be edited manually.
- GitHub Copilot receives both repository-wide and path-specific instructions.
- CODEOWNERS can enforce review boundaries for both instruction sources and runtime paths.
- Any new agent tool needs only a new adapter template and config entry.
- Functional projects generated by different tools are compared by behavior and
  acceptance criteria, not by source-code identity.
